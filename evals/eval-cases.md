## E1 · 地图作业：换电找车

**Prompt:**
```
使用 bos-design 设计一个换电员找车页面，地图上展示待换电车辆点位，支持按电量筛选，点击点位查看车辆简要信息，确认后导航前往。考虑定位失败和无待换电车辆的情况。
```

**覆盖场景:** 地图作业、点位交互、筛选、底部卡片、异常状态

**评判标准:**

| 类型 | 检查项 |
|------|--------|
| MUST | 输出包含问题定义（换电员、户外场景、找车目标） |
| MUST | 地图为主体视图，匹配 `display/map.md` 规范 |
| MUST | 筛选不遮挡地图主体操作区域 |
| MUST | 点击点位后展示车辆信息卡片（底部弹出或浮层） |
| MUST | 定位失败有引导恢复 |
| MUST | 无待换电车辆有 empty 状态 |
| SHOULD | 筛选使用 filter-bar 或地图顶部浮层 |
| SHOULD | 车辆信息卡片有明确的操作入口（导航/确认） |
| SHOULD | 考虑户外强光下的对比度和可读性 |

## E2 · 详情展示：车辆详情页

**Prompt:**
```
使用 bos-design 生成一个车辆详情页，展示车辆编号、车型、电量、锁车状态、最后定位时间和地址、所属网格、绑定换电柜、近7天骑行记录摘要。底部有"远程锁车"和"报修"操作。
```

**覆盖场景:** 内容详情、信息分组、状态展示、底部操作

**评判标准:**

| 类型 | 检查项 |
|------|--------|
| MUST | 使用了 content-detail page style 的结构思路 |
| MUST | 信息有分组（基础信息/位置信息/运营信息/骑行记录） |
| MUST | 状态字段使用 Tag 组件，匹配 `display/tag.md` |
| MUST | 底部操作栏匹配 `navigation/tool-bar.md` |
| MUST | "远程锁车"作为高风险操作有确认机制 |
| SHOULD | 有 NavBar 返回 |
| SHOULD | 电量有视觉化表达（颜色/进度条），不只是数字 |
| SHOULD | 骑行记录摘要不是全量数据，有"查看更多"入口 |

## E3 · 列表明细：任务列表

**Prompt:**
```
使用 bos-design 做一个运维任务列表页，支持按任务状态（待处理/进行中/已完成）切换，每条任务展示任务名称、类型标签、截止时间、负责人，支持搜索任务名称。
```

**覆盖场景:** 列表、Tab 切换、搜索、卡片、状态区分

**评判标准:**

| 类型 | 检查项 |
|------|--------|
| MUST | 使用了 object-search page style 的结构思路 |
| MUST | 状态切换使用 Tabs，匹配 `navigation/tab.md` |
| MUST | 搜索栏匹配 `input/search.md` |
| MUST | 列表项信息层级清晰（主标题/辅助信息/状态标签） |
| MUST | 空列表有 empty 状态 |
| MUST | 任务类型使用 Tag，匹配 `display/tag.md` |
| SHOULD | 有下拉刷新或上拉加载 |
| SHOULD | 截止时间临近有视觉提醒（颜色变化） |
| SHOULD | 列表项可点击进入详情 |

## E4 · 表单录入：规则创建

**Prompt:**
```
使用 bos-design 设计一个调度规则创建页面，需要填写规则名称、生效城市（多选）、触发条件（电量低于X%）、执行动作（派单/告警/锁车）、生效时间段。表单较长，需要分步或分组。
```

**覆盖场景:** 复杂表单、多选、条件配置、分组/分步、校验

**评判标准:**

| 类型 | 检查项 |
|------|--------|
| MUST | 使用了 data-entry page style 的结构思路 |
| MUST | 表单有分组或分步，不是一个无尽长表单 |
| MUST | 城市多选使用合适的选择器（Checkbox 列表或独立选择流程） |
| MUST | 触发条件有数值输入，匹配 `input/stepper.md` 或 Input 规范 |
| MUST | 有表单校验和错误提示 |
| MUST | 提交按钮匹配 `basic/button.md` |
| SHOULD | 执行动作使用 Checkbox 或 Radio 多选 |
| SHOULD | 生效时间段使用时间 Picker |
| SHOULD | 有"保存草稿"或返回时的未保存提醒 |

## E5 · 经营场景：运营表现看板

**Prompt:**
```
使用 bos-design 设计一个城市运营表现页面，展示今日订单完成率、车辆周转率、故障响应时长、用户投诉数，支持按日/周/月切换时间维度，数据异常时需要有提示。
```

**覆盖场景:** 数据看板、指标展示、时间维度切换、异常提示

**评判标准:**

| 类型 | 检查项 |
|------|--------|
| MUST | 使用了 operation-overview page style 的结构思路 |
| MUST | 核心指标有视觉优先级区分，不是四宫格平铺 |
| MUST | 时间维度切换有交互实现（Tabs 或 Segmented） |
| MUST | 数据异常有明确的视觉提示（颜色/图标/文案） |
| MUST | 没有装饰性渐变、发光等泛 AI 视觉习惯 |
| SHOULD | 指标有趋势方向表达（涨/跌箭头或颜色） |
| SHOULD | 有 loading/skeleton 过渡 |
| SHOULD | 指标可点击查看明细 |

## E6 · 增删查改：站点编辑

**Prompt:**
```
使用 bos-design 设计一个站点信息编辑页面，可修改站点名称、站点类型、容量上限、管理员，可删除站点（需二次确认）。页面从站点详情页进入。
```

**覆盖场景:** 编辑表单、预填数据、删除确认、页面关系

**评判标准:**

| 类型 | 检查项 |
|------|--------|
| MUST | 表单预填已有数据，不是空表单 |
| MUST | 删除操作有二次确认 Dialog，匹配 `feedback/dialog.md` |
| MUST | 保存按钮匹配 `basic/button.md` |
| MUST | 有表单校验 |
| MUST | 删除按钮视觉区分于保存（危险色或次级位置） |
| SHOULD | 未修改时保存按钮 disabled |
| SHOULD | 有 NavBar 标题标明"编辑站点" |
| SHOULD | 修改后未保存返回时有提醒 |

## E7 · 极简 Prompt 兜底

**Prompt:**
```
用 bos-design 做一个巡检页面
```

**覆盖场景:** 模糊需求下 skill 是否仍有 UX 判断而非套模板

**评判标准:**

| 类型 | 检查项 |
|------|--------|
| MUST | 没有直接套模板，先做了问题定义或合理假设 |
| MUST | 方案有明确的用户角色和场景上下文 |
| MUST | 页面结构有业务逻辑，不是通用 dashboard |
| MUST | 没有泛 AI 装饰（渐变卡片、浮动图形、空洞文案） |
| SHOULD | 考虑了巡检的特殊性（户外、逐项确认、拍照取证） |
| SHOULD | 有状态流转（待巡检 → 巡检中 → 已完成） |
