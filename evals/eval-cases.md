# Eval Cases

---

## 评判维度说明

每条 eval 的检查项分为两个维度：

- **目标达成** — 用户能否完成核心任务？信息是否引导决策？异常时能否恢复？流程是否顺畅？
- **规范实现** — 页面策略是否正确？组件 spec 是否匹配？token 和交互规则是否遵循？

两个维度同时通过才算合格。规范实现满分但用户目标达不成 = 不合格。

## E1 · 地图作业

**Prompt:**
```
使用 bos-design 设计一个换电员工作地图页面。地图上展示待换电车辆和换电柜点位，换电员需要快速判断哪些车最紧急（电量最低+距离最近），选中后导航前往。同时支持查看站点容量信息。考虑定位失败、区域内无车辆、网络离线的情况。
```

**用户目标:** 换电员在户外快速找到最该去换电的车，减少空跑

**评判标准:**

| 维度 | 类型 | 检查项 |
|------|------|--------|
| 目标达成 | MUST | 电量紧急程度有视觉区分，不需要逐个点开才能判断优先级 |
| 目标达成 | MUST | 从"看到车"到"去换电"的操作路径 ≤ 3步 |
| 目标达成 | MUST | 定位失败/无车辆/离线时用户知道发生了什么、能做什么 |
| 目标达成 | SHOULD | 距离信息辅助决策，不需要用户自己目测地图远近 |
| 目标达成 | SHOULD | 站点容量信息不干扰找车主流程 |
| 规范实现 | MUST | 地图为主体视图，匹配 `display/map.md` |
| 规范实现 | MUST | 筛选不遮挡地图核心操作区域 |
| 规范实现 | MUST | 异常状态匹配 `display/empty.md` |
| 规范实现 | SHOULD | 点位信息卡片使用 Popup/底部弹出 |
| 规范实现 | SHOULD | 图标使用 iconfont |

## E2 · 详情展示

**Prompt:**
```
使用 bos-design 生成一个任务详情页。任务包含：基本信息（编号、类型、优先级、截止时间）、关联对象（3辆车+1个站点）、执行要求（文字描述+检查项清单）、处理记录时间线（含多人流转）。底部有"开始处理"和"转派"操作。页面信息量大，需要让执行人快速抓到重点。
```

**用户目标:** 运维人员打开详情后 5 秒内知道"这个任务要我做什么、什么时候前做完、现在是什么状态"

**评判标准:**

| 维度 | 类型 | 检查项 |
|------|------|--------|
| 目标达成 | MUST | 首屏优先展示任务核心（做什么+截止时间+当前状态），不是从编号开始平铺 |
| 目标达成 | MUST | 信息分组有逻辑，用户能按需展开而非被迫全部阅读 |
| 目标达成 | MUST | 主操作"开始处理"视觉突出，不和"转派"平级 |
| 目标达成 | SHOULD | 关联对象可快速跳转，不是纯文本展示 |
| 目标达成 | SHOULD | 时间线能快速看到最新状态，不需要从头滚到尾 |
| 规范实现 | MUST | 使用 content-detail page style 结构 |
| 规范实现 | MUST | 时间线匹配 `display/timeline.md` |
| 规范实现 | MUST | 底部操作栏匹配 `navigation/tool-bar.md` |
| 规范实现 | SHOULD | 状态使用 Tag，匹配 `display/tag.md` |
| 规范实现 | SHOULD | 有 NavBar 和 Status Bar |

## E3 · 列表明细

**Prompt:**
```
使用 bos-design 做一个网格内车辆管理列表。运维主管需要从几百辆车中快速找到问题车辆：支持按状态（正常/故障/离线/低电量）筛选，按编号或车牌搜索，列表展示车辆核心状态。主管的核心诉求是"一眼看出哪些车需要处理"，而不是逐条检查。
```

**用户目标:** 主管快速定位需要处理的问题车辆，而非逐条浏览

**评判标准:**

| 维度 | 类型 | 检查项 |
|------|------|--------|
| 目标达成 | MUST | 问题车辆（故障/离线/低电量）有视觉突出，正常车辆弱化 |
| 目标达成 | MUST | 筛选和搜索能快速缩小范围，不需要多步操作 |
| 目标达成 | MUST | 列表信息层级清晰：一眼看到状态，不需要细读每行 |
| 目标达成 | MUST | 空结果时用户知道是"没有问题车"还是"搜索无匹配" |
| 目标达成 | SHOULD | 有数量统计，主管能感知问题规模 |
| 规范实现 | MUST | 使用 object-search page style 结构 |
| 规范实现 | MUST | 搜索匹配 `input/search.md` |
| 规范实现 | MUST | 筛选使用 filter-bar 或 Tab，不是自由输入 |
| 规范实现 | MUST | 状态使用 Tag，匹配 `display/tag.md` |
| 规范实现 | SHOULD | 有下拉刷新或上拉加载 |

## E4 · 表单录入

**Prompt:**
```
使用 bos-design 设计一个调度任务创建页面。需要填写：任务名称、任务类型（单选）、执行城市+街道+网格（三级联动）、指派人员（从列表选择）、截止时间、执行要求（多行文本）、附件上传。表单较长，填写人是运营主管，经常在移动中操作，需要减少输入负担。
```

**用户目标:** 主管在移动中快速创建任务，减少手动输入，避免漏填关键信息

**评判标准:**

| 维度 | 类型 | 检查项 |
|------|------|--------|
| 目标达成 | MUST | 能选的不让打字（类型、城市、人员都是选择而非输入） |
| 目标达成 | MUST | 必填项有明确标识，提交时校验能定位到具体字段 |
| 目标达成 | MUST | 三级联动不需要三次独立操作（级联或逐级自动展开） |
| 目标达成 | SHOULD | 表单有分组，用户能感知填写进度 |
| 目标达成 | SHOULD | 返回时有未保存提醒，防止误丢数据 |
| 规范实现 | MUST | 使用 data-entry page style 结构 |
| 规范实现 | MUST | 级联选择匹配 `input/cascader.md` 或多级 Picker |
| 规范实现 | MUST | 图片上传匹配 `input/image-upload.md` |
| 规范实现 | MUST | 按钮匹配 `basic/button.md` |
| 规范实现 | SHOULD | 时间选择使用 Picker/Calendar |

## E5 · 经营场景

**Prompt:**
```
使用 bos-design 设计一个城市运营日报页面。城市经理需要看今日核心指标（订单量、活跃车辆、故障率、调度完成率），快速判断哪些指标异常需要干预，支持按日/周切换对比趋势。经理的核心诉求是"今天有没有问题、问题在哪"，不是看全量数据。
```

**用户目标:** 城市经理 10 秒内判断今天运营是否正常，异常指标能直接定位到问题

**评判标准:**

| 维度 | 类型 | 检查项 |
|------|------|--------|
| 目标达成 | MUST | 异常指标有视觉突出（颜色/图标/位置），不需要逐个比对 |
| 目标达成 | MUST | 正常指标弱化，不和异常抢注意力 |
| 目标达成 | MUST | 时间切换能看到趋势变化，辅助判断"是突发还是持续恶化" |
| 目标达成 | SHOULD | 异常指标可下钻到明细（点击跳转或展开） |
| 目标达成 | SHOULD | 有对比基准（环比/同比/目标值），不是孤立数字 |
| 规范实现 | MUST | 使用 operation-overview page style 结构 |
| 规范实现 | MUST | 没有装饰性渐变、发光等泛 AI 视觉 |
| 规范实现 | MUST | 时间切换有交互实现 |
| 规范实现 | SHOULD | 有 loading/skeleton 过渡 |
| 规范实现 | SHOULD | 布局适合单手快速扫视 |

## E6 · 增删查改

**Prompt:**
```
使用 bos-design 设计一个站点编辑页面，从站点详情进入。可修改站点名称、类型、容量上限、管理员。可删除站点（删除后不可恢复）。编辑人是城市运营，需要确保不会误操作导致数据丢失。
```

**用户目标:** 运营能安全地修改站点信息，不会因为误触导致不可逆后果

**评判标准:**

| 维度 | 类型 | 检查项 |
|------|------|--------|
| 目标达成 | MUST | 删除有二次确认，且确认文案明确告知不可恢复 |
| 目标达成 | MUST | 删除操作在视觉和位置上与保存明确区分 |
| 目标达成 | MUST | 表单预填已有数据，用户知道在改什么 |
| 目标达成 | SHOULD | 未修改时保存按钮不可点击，避免无意义提交 |
| 目标达成 | SHOULD | 有修改内容未保存时返回提醒 |
| 规范实现 | MUST | 删除确认匹配 `feedback/dialog.md` |
| 规范实现 | MUST | 按钮匹配 `basic/button.md` |
| 规范实现 | MUST | 有表单校验 |
| 规范实现 | SHOULD | 有 NavBar 标题标明编辑态 |
| 规范实现 | SHOULD | 图标使用 iconfont |
