# 对象搜索（Object Search）

## 适用场景

用户在功能界面中点击搜索入口，跳转到独立搜索页面，查找目标对象。区别于列表页内的原地过滤搜索，这是一个完整的搜索场景页面。搜索的内容类型多样：人、车、站、城、计划、任务等。

## 需求目标

**用户目标：** 通过关键词快速找到目标对象（人、车、站、城、计划、任务等），查看关键信息或进入操作。

**产品目标：** 提供高效的跨层级、跨类型搜索能力，让用户不必逐级导航就能直达目标；同时在搜索结果中暴露关键业务指标，辅助快速判断和决策。

## 首屏优先级

- **必须出现**：搜索框（自动聚焦 + 键盘弹起）、历史记录（如有）
- **搜索后必须出现**：结果统计 + 第一张结果卡片

## 框架设计

页面是一个**状态驱动的单页面**，根据搜索进度展示不同内容：

- 固定区：搜索框（始终置顶）
- 滚动区：根据状态切换内容（历史记录 / 结果卡片 / 空态）
- 无固定底部操作栏（搜索页的操作就是搜索本身）

状态流转：初始空白 → 有历史记录 → 输入中（键盘弹起）→ 有结果 → 无结果

## 信息组织

### 信息层级

1. **最高优先级 — 搜索框**：置顶且自动聚焦，进入即可输入，零操作成本。
2. **第二优先级 — 搜索结果**：结果承载用户确认目标所需的信息，尽量让用户不用点进去就能完成第一轮判断。
3. **辅助信息 — 历史记录**：用 Tag 形式横向排列，节省纵向空间，一眼可见多条记录。
4. **辅助信息 — 结果统计**：右对齐的「找到 X 个 XX」，快速告知结果数量。

### 组织范式

#### 搜索框自动聚焦，零成本启动

进入页面即弹出键盘，光标在搜索框。不需要用户再点一次。搜索页的唯一目的就是搜索，不要在启动环节加摩擦。

#### 历史记录用 Tag 而非列表

横向排列的 Tag 比纵向列表更节省空间，在键盘弹起的有限空间里能展示更多条历史。点击即填入搜索词并触发搜索。

#### 结果形态跟随内容与决策密度

搜索结果先看用户识别目标所需的信息量。若对象需要同时展示状态、基础统计或业务指标（如完成率、作业量、运维人数），可用卡片承载多层信息；若结果只需快速扫名称、编号、状态或少量字段即可判断，使用更紧凑的列表形态更合适。

#### 结果内容跟随对象类型变化

搜街道可展示运营指标，搜车可展示车辆状态，搜人可展示出勤信息。先保持当前结果形态内部的信息层级一致，再让内容字段跟随对象类型变化。

#### 键盘在有结果后收起

有结果时让出空间展示结果卡片。用户需要修改搜索词时点击搜索框重新唤起键盘。

## 交互与状态

### 关键交互

- 自动聚焦：进入页面即弹出键盘，光标在搜索框
- 历史记录：点击 Tag 直接填入搜索词并触发搜索
- 一键清空：清除所有历史记录
- 搜索触发：输入后回车或实时搜索展示结果
- 结果项点击：进入对应对象的详情或操作页
- 键盘收起：有结果后键盘自动收起或用户下滑收起

### 状态覆盖

- 初始空白：搜索框聚焦 + 键盘弹起
- 有历史记录：展示历史 Tag
- 搜索中：加载态
- 有结果：结果卡片列表
- 无结果：空态插图 + 「未找到符合条件的 XX」
- 到底：「已经到底」提示

## Do / Don't

- **Do**：进入即聚焦搜索框，不要让用户再点一次
- **Do**：搜索结果展示关键业务数据，让用户不用点进去就能判断
- **Do**：保留搜索历史，运维人员经常搜同一个对象
- **Do**：结果统计明确告知数量，0 结果时给出明确反馈
- **Do**：按搜索结果的识别成本和决策信息量选择紧凑列表或卡片承载
- **Do**：同一结果形态内保持层级一致，内容字段跟随对象类型变化
- **Don't**：不要在用户需要状态或业务数据才能判断时，只显示对象名称
- **Don't**：不要用无限滚动，搜索结果通常不多，一次性展示即可
- **Don't**：不要在空态只放文字，插图能更快传达「没有结果」的信息
- **Don't**：不要把独立搜索页和列表内过滤搜索混为一谈，两者交互模式不同
