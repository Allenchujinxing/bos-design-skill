# Page Style Library — 分析约束

> 本文件定义了分析设计稿时的维度和学习要求。每次收到新的设计稿时，必须按照以下约束进行解读和总结。

## 分析维度

收到界面设计稿时，从以下维度解读：

1. **适用场景** — 这个页面什么时候用，用户是谁，在什么任务里打开
2. **需求目标** — 用户进来最想完成什么；系统希望用户完成什么、看见什么、避免什么风险
3. **首屏优先级** — 首屏必须出现哪些信息，哪些可以下滑或收起
4. **框架设计** — 页面的骨架是什么，固定区和滚动区怎么划分，区域之间的关系
5. **信息组织** — 什么信息最重要放最突出，什么是辅助的，什么可以收起；信息之间的关系怎么表达（分组、嵌套、联动、前置引导等）
6. **交互与状态** — 关键交互（选择、提交、筛选、弹层、确认等）；状态覆盖（默认、加载、空、错误、权限不足、提交成功/失败等）
7. **Do / Don't** — 这个页面类型里一定要做什么，绝对不要做什么

## 学习要求

### 重点提炼设计思路，而非罗列组件

- 聚焦在**框架设计、信息层级、组织范式**这三个维度
- 提炼的是「面对什么场景时，应该怎么组织信息、怎么布局、怎么设计交互流程」
- 组件只是实现手段，不是 page style 的核心内容
- 不要列组件清单，要写设计决策和背后的原因

### 信息布局策略必须写明

每个 page style 必须包含信息布局策略，说明：
- 为什么这样分区
- 为什么这个信息放在这个位置
- 信息之间的关系如何通过布局表达

### 基于用户场景来设计

- 表单的设计是基于用户场景来的，不是基于组件能力
- 不同场景下同一种页面类型的具体内容会不同
- page style 记录的是设计策略和范式，不是固定模板

### 区分交互模式和内容设计

- 底部弹层是交互模式，里面的内容设计根据场景变化
- 独立页面和弹层是两种不同的交互模式，不要混淆
- 列表内原地过滤和跳转独立搜索页是两种不同的搜索模式

## 输出格式

每个 page style 文件应包含：
- 适用场景
- 需求目标（用户目标 + 产品目标）
- 首屏优先级
- 框架设计（骨架、固定区/滚动区）
- 信息组织（层级排序 + 组织范式 + 设计策略与原因）
- 交互与状态（关键交互 + 状态覆盖）
- Do / Don't
